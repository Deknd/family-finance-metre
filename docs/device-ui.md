# UI для физического устройства на ESP32 + E-Ink 296x160

## 1. Назначение UI

UI устройства должен за несколько секунд отвечать на вопрос:

- финансовая ситуация семьи сейчас нормальная, тревожная или рискованная;
- как выглядят 3-4 главные метрики без открытия приложения.

Это не аналитический экран и не мини-кабинет. Это компактная приборная панель.

## 2. Ограничения дисплея

Параметры дисплея:

- 296 x 160 px
- 2 оттенка серого
- e-ink
- полное обновление около 23 секунд

Следствия для UI:

- один экран должен быть очень простым;
- нельзя перегружать интерфейс мелким текстом;
- нельзя рассчитывать на частые перерисовки;
- нужно показывать только крупные и стабильные показатели;
- основной сценарий должен помещаться на один экран.

## 3. Принципы интерфейса

- один экран = одна главная мысль;
- на экране не больше 4 ключевых метрик;
- главный акцент на статусе и крупных числах;
- текст короткий и читаемый с расстояния;
- второстепенная информация уходит вниз экрана;
- UI должен одинаково хорошо смотреться в черно-белом виде.

## 4. Состав UI для MVP

Для MVP основной экран показывает только:

- статус;
- доход в месяц;
- расходы в месяц;
- кредитную нагрузку;
- подушку в месяцах;
- время последнего обновления.

Это базовый и достаточный состав для первой версии устройства.

## 5. Основной экран MVP

### 5.1. Структура

Экран делится на 3 зоны:

- верхняя зона: статус и короткая причина;
- центральная зона: 4 основные метрики;
- нижняя зона: время обновления.

### 5.2. Текстовый макет

```text
+----------------------------------+
| СТАТУС: ВНИМАНИЕ                 |
| Подушка ниже комфортной зоны     |
|----------------------------------|
| Доход            210 000         |
| Расходы           90 000         |
| Кредиты             27%          |
| Подушка            2.0 мес       |
|----------------------------------|
| Обновлено        14.03 18:30     |
+----------------------------------+
```

## 6. Описание элементов интерфейса

### 6.1. Статус

Статус занимает самую заметную область вверху.

Варианты:

- `СТАТУС: НОРМА`
- `СТАТУС: ВНИМАНИЕ`
- `СТАТУС: РИСК`

Под статусом допускается одна короткая строка причины.

Примеры:

- `Подушка ниже комфортной зоны`
- `Кредитная нагрузка выше комфортной`

MVP policy для UI:

- `СТАТУС: НОРМА`, если кредитная нагрузка ниже `30%`
  и подушка при ненулевых расходах не меньше `3.00` месяцев
- `СТАТУС: ВНИМАНИЕ`, если кредитная нагрузка от `30%` до `49.99%`
  или подушка при ненулевых расходах ниже `3.00`, но не ниже `1.00` месяца
- `СТАТУС: РИСК`, если кредитная нагрузка от `50%`
  или подушка при ненулевых расходах ниже `1.00` месяца
- если одновременно срабатывают несколько причин,
  UI должен показывать ту причину, которую прислал сервер;
  при равной тяжести сервер отдает причину по подушке раньше причины по кредитам

### 6.2. Блок метрик

В центральной части экрана размещаются 4 строки:

- `Доход`
- `Расходы`
- `Кредиты`
- `Подушка`

Требования:

- подписи короткие и одинакового формата;
- значения выровнены по правому краю;
- у процентов и месяцев единицы измерения показываются явно;
- у денежных значений можно не писать валюту в каждой строке, если она едина для устройства.

### 6.3. Время обновления

В нижней части показывается:

- `Обновлено 14.03 18:30`

Это помогает пользователю понимать свежесть данных без лишних деталей.

## 7. Визуальная иерархия

Рекомендации по иерархии:

- статус: самый крупный и контрастный элемент;
- причина статуса: маленький или средний шрифт;
- значения метрик: средний или крупный шрифт;
- подписи метрик: меньше значений;
- время обновления: самый спокойный визуальный элемент.

Для e-ink хорошо подойдут:

- черная плашка под статус;
- тонкие горизонтальные разделители;
- минимум декоративных элементов;
- отсутствие сложных графиков и иконок в MVP.

## 8. Требования к читаемости

- не больше 30-40 символов в строке;
- не больше 6-7 текстовых строк с важной информацией;
- значения должны читаться с расстояния;
- нельзя использовать длинные пояснения;
- каждая метрика должна помещаться в одну строку.

## 9. Итоговая рекомендация

Для первой версии устройства нужен один основной экран без переключений и без перегрузки интерфейса.

Лучший формат для MVP:

- один статус;
- одна короткая причина;
- четыре основные метрики;
- одна строка времени обновления.

Примечание на будущее:

- в MVP причина статуса приходит уже готовой с сервера;
- позже правила и текст причины могут формироваться через `n8n` или LLM-агент,
  но UI по-прежнему должен работать с теми же полями `status`, `status_text`, `status_reason`.
