# План реализации MVP сервера

## 1. Назначение

Этот документ описывает пошаговый план реализации MVP сервера.

План построен так, чтобы:

- каждая задача закрывала один небольшой кусок функционала;
- задачи можно было брать в работу по порядку;
- итоговый сервер покрывал сценарии из текущей документации.

## 2. Границы MVP

В этот план входят:

- прием данных от `n8n` после опроса пользователя;
- расчет семейных агрегатов;
- выдача данных физическому устройству;
- логика запуска LLM-агента после зарплатных событий;
- хранение данных в БД.

В этот план не входят:

- UI админки;
- полноценное управление пользователями;
- сложная аналитика;
- праздничные календари;
- ручное редактирование финансов через веб-интерфейс.

## 3. Предположение для реализации

Для MVP предполагается:

- сервер реализуется на Java;
- сервер использует Spring Boot 4.0.3;
- минимально поддерживаемая версия Java для Spring Boot 4.0.3 - Java 17, но для проекта рекомендуется Java 21 LTS;
- сервер использует PostgreSQL;
- семьи, члены семьи, устройства и зарплатные правила заводятся через seed/скрипт или внутренний admin-script;
- публичных CRUD endpoint'ов для управления этими сущностями в MVP не будет.

Основание для версии:

- по официальной документации Spring на 14 марта 2026 года стабильная версия Spring Boot - `4.0.3`;
- официальные system requirements для Spring Boot 4.0.3 требуют Java 17+.

## 5. Архитектурный подход для сервера

Актуальная архитектура сервера зафиксирована в отдельном документе:

- `docs/server/architecture.md`

Базовый подход:

- сервер строится как модульный монолит;
- предметная модель и расчеты лежат в `core`;
- прикладные потоки данных разделены по пакетам `flow.intake`, `flow.dashboard`, `flow.collection`;
- общие технические компоненты лежат в `shared`.

Ключевые правила:

- `core` не зависит от `flow.*`;
- разные `flow` не зависят друг от друга напрямую;
- legacy-структура верхнего уровня `controller/service/repository/entity/dto/...`
  больше не используется как точка роста.

Это дает:

- понятные архитектурные границы;
- устойчивость к дальнейшему росту MVP;
- изоляцию бизнес-потоков;
- возможность развивать сервер без смешивания транспортного и предметного кода.

## 6. Очередность задач

## Задача 01. Поднять каркас серверного приложения

Что сделать:

- создать проект через Spring Initializr;
- подключить Java 21;
- подключить Spring Boot 4.0.3;
- подключить Maven wrapper;
- добавить `spring-boot-starter-webmvc`;
- добавить `spring-boot-starter-actuator`;
- добавить health endpoint через Actuator.

Результат:

- сервер запускается локально;
- есть базовая точка входа для дальнейшей реализации.

## Задача 02. Подключить конфигурацию окружения

Что сделать:

- настроить `application.yml`;
- добавить `application-local.yml`;
- добавить `@ConfigurationProperties` для конфигурации БД, токенов и интеграций;
- вынести URL webhook для `n8n`;
- вынести настройки scheduler;
- вынести timezone приложения.

Результат:

- сервер конфигурируется без хардкода.

## Задача 03. Сделать миграцию для справочных таблиц

Что сделать:

- подключить Flyway;
- создать первую миграцию;
- создать таблицы `families`, `family_members`, `devices`.

Результат:

- в БД можно хранить семьи, людей и физические устройства.

## Задача 04. Сделать миграцию для зарплатных правил

Что сделать:

- создать таблицу `member_payroll_schedules`;
- заложить поддержку нескольких зарплатных событий на одного человека;
- заложить `fixed_day_of_month` и `last_day_of_month`;
- заложить `trigger_delay_days`.

Результат:

- сервер может хранить расписание выплат для каждого члена семьи.

## Задача 05. Сделать миграцию для интеграционных и расчетных таблиц

Что сделать:

- создать `llm_collection_requests`;
- создать `finance_submissions`;
- создать `member_finance_snapshots`;
- создать `family_dashboard_snapshots`.

Результат:

- база готова для приема данных, пересчета и выдачи агрегатов.

## Задача 06. Сделать seed или bootstrap-скрипт для начальных данных

Что сделать:

- добавить способ завести тестовую семью;
- добавить членов семьи;
- добавить устройство;
- добавить зарплатные правила.

Рекомендуемый способ:

- SQL seed для локальной среды;
- или отдельный `CommandLineRunner`, который включается только в `local` profile.

Результат:

- можно быстро развернуть локальный стенд и тестировать сервер end-to-end.

## Задача 07. Реализовать базовый слой доступа к БД

Что сделать:

- подключить Spring Data JPA;
- описать JPA entity для основных таблиц;
- создать `JpaRepository` для основных сущностей;
- вынести транзакционные сервисы;
- определить стратегию работы с `jsonb` полями.

Результат:

- прикладная логика не смешана с сырыми SQL-вызовами.

## Задача 08. Реализовать middleware аутентификации

Что сделать:

- сделать простой auth layer без Spring Security OAuth;
- реализовать проверку `X-Device-Token` для устройства;
- реализовать проверку `X-API-Key` для `n8n`;
- вынести проверки в `OncePerRequestFilter` или `HandlerInterceptor`;
- добавить единый формат ошибок `401`.

Результат:

- сервер различает доверенные вызовы устройства и интеграции.

## Задача 09. Реализовать валидацию входного payload от `n8n`

Что сделать:

- описать схему `POST /api/v1/intake/user-finance-data`;
- валидировать обязательные поля;
- валидировать денежные значения и период;
- использовать Jakarta Validation annotations;
- добавить глобальный `@RestControllerAdvice`;
- отдавать `422` при невалидном теле.

Результат:

- сервер принимает только корректные данные по контракту.

## Задача 10. Реализовать защиту от дублей для intake endpoint

Что сделать:

- проверять `external_submission_id`;
- возвращать `409`, если такой payload уже был обработан.

Результат:

- повторная отправка из `n8n` не создает дубликаты.

## Задача 11. Реализовать endpoint приема данных от агента

Что сделать:

- поднять `POST /api/v1/intake/user-finance-data`;
- после успешной валидации сохранять payload как входящее событие.

Spring-реализация:

- `@RestController`
- request DTO
- response DTO
- сервис intake-обработки

Результат:

- есть рабочий endpoint приема данных от `n8n`.

## Задача 12. Реализовать сохранение в `finance_submissions`

Что сделать:

- сохранять нормализованные поля;
- сохранять `raw_payload`;
- сохранять связь с `request_id` и `llm_collection_request_id`, если они есть.

Spring-реализация:

- сервисный метод в `@Transactional`;
- entity + repository для `finance_submissions`.

Результат:

- сервер хранит полный факт приема данных и может его повторно обработать.

## Задача 13. Реализовать upsert в `member_finance_snapshots`

Что сделать:

- по `member_id + period_year + period_month` пересчитывать месячный снимок;
- суммировать `monthly_income` по всем `finance_submission` за период;
- использовать самую свежую `finance_submission` для `monthly_expenses`, `monthly_credit_payments`, `liquid_savings`, `source_submission_id` и `collected_at`.

Spring-реализация:

- отдельный application service;
- пересчет snapshot внутри одной транзакции.

Результат:

- у сервера всегда есть актуальные месячные данные по каждому человеку с корректной суммой всех выплат за месяц.

## Задача 14. Реализовать сервис агрегации семейных метрик

Что сделать:

- суммировать доходы членов семьи;
- суммировать расходы членов семьи;
- суммировать кредитные платежи;

Spring-реализация:

Результат:

- сервер умеет из персональных данных собрать семейный срез.

## Задача 15. Реализовать сервис расчета dashboard-метрик

Что сделать:

- посчитать `credit_load_percent`;
- посчитать `emergency_fund_months`;
- подготовить данные для физического экрана.

Spring-реализация:

- отдельный `DashboardCalculationService`;
- использовать `BigDecimal` для процентов и промежуточных вычислений.

Результат:

- агрегированные значения готовы к показу на устройстве.

## Задача 16. Реализовать сервис расчета статуса семьи

Что сделать:

- определить `normal`, `warning`, `risk`;
- вычислить `status_text`;
- вычислить `status_reason`.

Spring-реализация:

- вынести правила в отдельный policy/service класс, а не держать их внутри controller.

Результат:

- сервер умеет отдавать не только цифры, но и итоговый статус для UI.

## Задача 17. Реализовать сохранение `family_dashboard_snapshots`

Что сделать:

- после каждого нового intake пересчитывать семейный dashboard;
- обновлять или создавать снимок за период.

Spring-реализация:

- orchestration service, который после intake вызывает:
- пересчет `member_finance_snapshots`
- перерасчет family aggregate
- сохранение `family_dashboard_snapshots`

Результат:

- данные для устройства хранятся в готовом виде.

## Задача 18. Реализовать endpoint для физического устройства

Что сделать:

- поднять `GET /api/v1/device/dashboard`;
- находить устройство по токену;
- находить связанную семью;
- отдавать `family_dashboard_snapshot`
  за последний доступный расчетный период семьи;
- возвращать `404`, если данных еще нет.

Spring-реализация:

- `@RestController`
- response DTO, совпадающий с `docs/server/api-spec.md`

Результат:

- устройство может получать готовую информацию для экрана.

## Задача 19. Добавить обновление `last_seen_at` устройства

Что сделать:

- при успешном ответе `GET /api/v1/device/dashboard`
  обновлять `devices.last_seen_at`;
- не обновлять `devices.last_seen_at`, если запрос завершился ошибкой.

Spring-реализация:

- выполнять update после успешного формирования payload для
  `GET /api/v1/device/dashboard`, а не сразу после авторизации устройства.

Результат:

- сервер знает, когда устройство последний раз было на связи.

## Задача 20. Реализовать сервис расчета зарплатных событий

Что сделать:

- считать номинальную дату выплаты;
- считать фактическую дату выплаты;
- учитывать `fixed_day_of_month`;
- учитывать `last_day_of_month`;
- для `fixed_day_of_month` в коротком месяце использовать последний календарный день месяца;
- переносить субботу и воскресенье на предыдущую пятницу.

Spring-реализация:

- отдельный pure Java service без зависимости от web-слоя;
- использовать `java.time.LocalDate`.

Результат:

- сервер умеет корректно определить, когда должен стартовать опрос.

## Задача 21. Реализовать ежедневный планировщик опросов

Что сделать:

- запускать ежедневную джобу;
- проходить по активным `member_payroll_schedules`;
- вычислять `scheduled_trigger_date`;
- определять, для кого сегодня надо запускать опрос.

Spring-реализация:

- включить `@EnableScheduling`;
- использовать `@Scheduled`;
- учитывать timezone семьи при вычислении текущей даты.

Результат:

- сервер автоматически находит зарплатные события, требующие запуска агента.

## Задача 22. Реализовать создание `llm_collection_requests`

Что сделать:

- создавать запись перед вызовом `n8n`;
- сохранять `payroll_schedule_id`;
- сохранять `nominal_payroll_date`, `effective_payroll_date`, `scheduled_trigger_date`;
- защищаться от дублей по одному зарплатному событию.

Spring-реализация:

- создание записи и вызов клиента разделить по сервисам;
- защита от дублей через уникальный индекс + обработка `DataIntegrityViolationException`.

Результат:

- каждый запуск опроса трассируется и не дублируется.

Spring-реализация:

Результат:

- сервер умеет реально инициировать сбор данных через LLM-агента.

Spring-реализация:

- различать сетевые ошибки и бизнес-ошибки ответа.

Результат:

- у сервера есть прозрачный статус каждого запуска опроса.

## 8. Что получится в конце MVP

После выполнения всех задач сервер сможет:

- хранить семьи, людей, устройства и зарплатные правила;
- по расписанию запускать `n8n` для нужного человека;
- принимать данные после опроса;
- пересчитывать семейные агрегаты;
- отдавать готовый dashboard физическому устройству.
