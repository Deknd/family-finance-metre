# Архитектура сервера

## 1. Назначение

Этот документ фиксирует актуальные архитектурные правила серверной части проекта.

Цель архитектуры:

- разделять код по смыслу и назначению, а не только по техническому слою;
- держать предметную логику отдельно от прикладных потоков данных;
- не допускать перемешивания сценариев `server -> n8n`, `n8n -> server` и `server -> device`;
- сохранять простоту модульного монолита без преждевременного усложнения до multi-module или full hexagonal architecture.

## 2. Базовая модель

Сервер строится как модульный монолит с тремя верхнеуровневыми зонами:

- `core` — предметная модель и бизнес-логика;
- `flow` — прикладные потоки данных;
- `shared` — общие технические компоненты.

В корне `src/main/java/com/deknd/familyfinancemetre` рабочими пакетами считаются только:

- `core`
- `flow`
- `shared`

Старые верхнеуровневые пакеты `controller`, `service`, `repository`, `entity`, `dto`, `security`, `config`, `validation`, `exception`
считаются legacy-структурой и не должны использоваться для нового кода.

## 3. Пакетная структура

### 3.1. `core`

`core` хранит то, что относится к сущностям и предметным правилам проекта.

Типичный состав:

- JPA entity;
- enum и JPA converter;
- repository;
- сервисы расчета и policy;
- value-like классы, используемые только внутри предметной логики.

Внутри `core` код группируется по предметным частям:

- `core.family`
- `core.device`
- `core.payroll`
- `core.collection`
- `core.snapshot`
- `core.common`

Примеры:

- `core.payroll` — расчет payroll-событий;
- `core.snapshot` — пересчет submission, snapshot и status policy;
- `core.collection` — сущность, repository и lifecycle policy для `llm_collection_requests`.

### 3.2. `flow`

`flow` хранит прикладные сценарии прохождения данных через систему.

Каждый поток содержит только свой web-слой, DTO и orchestration-логику.

Актуальные потоки:

- `flow.intake` — сценарий `n8n -> server`;
- `flow.dashboard` — сценарий `server -> device`;
- `flow.collection` — сценарий `server -> n8n`.

Типичный состав потока:

- `dto`
- `web`
- `service`
- `exception`

Допускается добавлять внутри потока дополнительные подпакеты, если они относятся только к этому сценарию:

- `client`
- `scheduler`
- `mapper`
- `model`

### 3.3. `shared`

`shared` хранит общие технические компоненты, которые не выражают бизнес-смысл конкретной сущности или потока.

Типичный состав:

- `shared.config`
- `shared.security`
- `shared.validation`
- `shared.web`

Важно:

- в `shared` нельзя складывать предметные правила;
- `shared` не должен становиться «свалкой общего кода»;
- если логика относится к домену семьи, payroll, snapshot или collection, ей место в `core`, а не в `shared`.

## 4. Правила зависимостей

### 4.1. Главные правила

- `core` не зависит от `flow.*`;
- `flow.*` может зависеть от `core` и `shared`;
- прямые зависимости между разными `flow`-пакетами запрещены;
- `shared` не должен зависеть от конкретного прикладного потока.

Практический смысл:

- intake-поток может вызвать пересчет snapshot через `core.snapshot`;
- dashboard-поток может читать snapshot через `core.snapshot`;
- но intake-поток не должен вызывать dashboard-поток напрямую;
- payroll/collection-поток не должен ходить в intake-поток за orchestration-логикой.

### 4.2. Что считается нарушением

Нарушением считается любой из сценариев:

- импорт класса из `flow.dashboard` внутри `flow.intake`;
- импорт класса из `flow.intake` внутри `flow.collection`;
- импорт класса из любого `flow.*` внутри `core.*`;
- добавление нового production-класса в legacy-пакеты верхнего уровня.

## 5. Правила размещения кода

### 5.1. Что класть в `core`

Код идет в `core`, если он отвечает на вопрос:

- что является сущностью системы;
- как считается бизнес-правило;
- как хранится и пересчитывается доменная модель;
- какие значения и статусы считаются корректными в предметной области.

Примеры:

- `FamilyEntity`
- `MemberPayrollScheduleEntity`
- `PayrollEventCalculationService`
- `FamilyDashboardStatusPolicy`
- `MemberFinanceSnapshotRecalculationService`
- lifecycle-сервис статусов `llm_collection_requests`,
  общий для `flow.collection` и `flow.intake`

### 5.2. Что класть в `flow`

Код идет в конкретный `flow`, если он отвечает на вопрос:

- как проходит один конкретный сценарий интеграции;
- какой HTTP endpoint обслуживает этот сценарий;
- какой request/response DTO нужен только этому потоку;
- как выглядит orchestration именно этого канала.

Примеры:

- `flow.intake.web.UserFinanceIntakeController`
- `flow.intake.dto.UserFinanceIntakeRequest`
- `flow.dashboard.web.DeviceDashboardController`
- `flow.dashboard.dto.DeviceDashboardResponse`

### 5.3. Что класть в `shared`

Код идет в `shared`, если он:

- одинаково полезен нескольким потокам;
- относится к инфраструктуре, а не к бизнес-смыслу;
- не выражает отдельную предметную подмодель.

Примеры:

- security filters;
- `@ConfigurationProperties`;
- custom validation annotations;
- общий формат web-ошибок.

## 6. Правила для тестов

Тесты должны зеркалить архитектуру production-кода.

Это означает:

- тесты `core` лежат в `src/test/java/.../core/...`;
- тесты `flow.intake` лежат в `src/test/java/.../flow/intake/...`;
- тесты `flow.dashboard` лежат в `src/test/java/.../flow/dashboard/...`;
- тесты `shared.security` лежат в `src/test/java/.../shared/security/...`.

Так тестовая структура показывает границы системы так же явно, как и production-код.

## 7. Архитектурная защита

Архитектурные правила дополнительно проверяются через `ArchUnit`.

Проверяется:

- запрет зависимостей `core -> flow`;
- запрет зависимостей между разными `flow`;
- запрет возврата к legacy-пакетам верхнего уровня.

Если новая задача требует изменить эти правила, сначала нужно обновить этот документ,
а затем уже осознанно менять архитектурный тест.

## 8. Правило развития

При добавлении нового кода сначала нужно ответить на два вопроса:

1. Это предметная логика или прикладной сценарий?
2. Если это сценарий, к какому именно потоку он относится?

После этого:

- предметная логика идет в `core`;
- код сценария идет в соответствующий `flow`;
- общая инфраструктура идет в `shared`.

Если новый код не удается уверенно отнести ни к одной из зон,
это сигнал остановиться и уточнить архитектурную границу до начала реализации.
