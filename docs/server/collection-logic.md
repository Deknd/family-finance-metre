# Логика сервера для запуска опросов после зарплаты

## 1. Назначение

Этот документ описывает фактическую серверную логику, по которой создаются задачи на опрос пользователя после зарплатных событий.

Документ не описывает API и БД полностью. Он фиксирует правила:

- расчета payroll-даты;
- вычисления даты запуска опроса;
- работы ежедневного scheduler;
- создания и lifecycle `llm_collection_requests`;
- валидации ответа `n8n`.

## 2. Основная идея

Для каждого члена семьи сервер хранит одно или несколько правил зарплатных выплат.

Примеры:

- у одного человека зарплата приходит 16 числа и в последний день месяца;
- у другого человека зарплата приходит 5 и 25 числа.

Для каждого такого события сервер:

- вычисляет номинальную дату выплаты;
- вычисляет фактическую дату выплаты;
- вычисляет дату запуска опроса;
- в нужный день создает `llm_collection_request`;
- вызывает webhook в `n8n`;
- не допускает повторного запуска для того же payroll-события.

Каждый запуск опроса соответствует одной конкретной выплате, а не месяцу целиком.

## 3. Входные данные для планировщика

Планировщик опирается на таблицу:

- `member_payroll_schedules`

Каждая запись в ней описывает одно правило выплаты:

- фиксированный день месяца;
- или последний день месяца.

В обработку попадают только:

- активные `member_payroll_schedules`;
- активные `family_members`;
- семьи со статусом `active`.

## 4. Расчет даты выплаты

### 4.1. Номинальная дата выплаты

Сначала сервер вычисляет номинальную дату выплаты.

Правила:

- если `schedule_type = fixed_day_of_month`, берется соответствующий день месяца;
- если в таком месяце нужного дня нет, берется последний календарный день месяца;
- если `schedule_type = last_day_of_month`, берется последний календарный день месяца.

Примеры:

- `16` -> `2026-03-16`;
- `31` для апреля 2026 -> `2026-04-30`;
- `last_day_of_month` для марта -> `2026-03-31`;
- `last_day_of_month` для февраля 2026 -> `2026-02-28`.

### 4.2. Фактическая дата выплаты

После этого сервер проверяет, является ли номинальная дата выходным днем.

Текущее правило MVP:

- выходные - суббота и воскресенье;
- официальные праздники пока не учитываются;
- если дата попадает на выходной, она переносится на предыдущую пятницу.

### 4.3. Примеры переноса

- если дата выпала на субботу, перенос на пятницу;
- если дата выпала на воскресенье, перенос на пятницу.

Другие варианты переноса в текущем MVP не используются.

## 5. Расчет даты запуска опроса

После определения `effective_payroll_date` сервер рассчитывает:

- `scheduled_trigger_date = effective_payroll_date + trigger_delay_days`

Если `trigger_delay_days = 1`, опрос стартует на следующий день после фактической выплаты.

Пример:

- фактическая выплата: `2026-03-16`;
- `trigger_delay_days = 1`;
- запуск опроса: `2026-03-17`.

## 6. Как определяется расчетный период

Сервер использует простое правило:

- `period_year` и `period_month` совпадают с годом и месяцем `effective_payroll_date`.

Пример:

- зарплата пришла `2026-03-31`;
- опрос стартовал `2026-04-01`;
- данные относятся к периоду `2026-03`.

Это важно для выплат на границе месяца.

## 7. Работа scheduler

Текущая схема:

1. Планировщик запускается по cron из `app.scheduler.payroll-collection`.
2. Для каждого правила сервер берет текущую дату именно в таймзоне семьи.
3. Для каждой такой семейной даты сервер проверяет два candidate month:
   текущий месяц и предыдущий месяц.
4. Для каждого candidate month вычисляются `nominal_payroll_date`, `effective_payroll_date` и `scheduled_trigger_date`.
5. Если `scheduled_trigger_date` совпадает с текущей датой в таймзоне семьи, сервер пытается создать запрос на сбор данных.
6. Если запрос уже существует для того же payroll-события, повторный запуск не выполняется.

Проверка предыдущего месяца нужна для сценария на границе месяца:

- зарплата пришла `2026-03-31`;
- опрос должен стартовать `2026-04-01`;
- в день `2026-04-01` сервер все еще должен увидеть payroll-событие марта.

## 8. Защита от дублей

Для одного payroll-события сервер считает уникальным сочетание:

- `payroll_schedule_id`;
- `effective_payroll_date`.

Это означает:

- для одной и той же зарплаты опрос можно запустить только один раз.

На уровне БД это защищается уникальным индексом,
а на уровне сервиса повторное создание подавляется как already-created сценарий.

## 9. Создание `llm_collection_requests`

Перед вызовом `n8n` сервер создает запись в `llm_collection_requests` со статусом `pending`.

В записи сохраняются:

- `request_id` как основной correlation id цепочки;
- `period_year` и `period_month`;
- `reason = day_after_salary`;
- `requested_fields`;
- payroll-датировки;
- `triggered_at`;
- полный `request_payload`, который будет отправлен в `n8n`.

Для payroll-triggered потока сервер сам генерирует `request_id` как UUID-строку.

На стороне `n8n` это значение должно использоваться как непрозрачный correlation id
и возвращаться без изменений.

## 10. Вызов `n8n` и валидация ответа

После создания `pending`-запроса сервер вызывает webhook `n8n`
с телом из `llm_collection_requests.request_payload`.

Вызов считается успешным только если одновременно выполнены все условия:

- `n8n` вернул `HTTP 202 Accepted`;
- в теле ответа `status = accepted`;
- `request_id` в ответе совпадает с отправленным `request_id`;
- `workflow_run_id` присутствует и не пустой.

Если эти условия выполнены, сервер:

- переводит запись из `pending` в `accepted`;
- сохраняет `accepted_at`;
- сохраняет `workflow_run_id`;
- сохраняет `response_payload`.

Если HTTP-вызов завершился ошибкой или ответ не прошел contract validation, сервер:

- переводит запись из `pending` в `failed`;
- сохраняет `response_payload`, если оно есть;
- сохраняет человекочитаемое `error_message`.

## 11. Lifecycle `llm_collection_requests`

Текущий server-side lifecycle фиксирован так:

- `pending -> accepted`;
- `pending -> failed`;
- `accepted -> completed`.

Статус `cancelled` присутствует в модели и БД как зарезервированный,
но текущая server-side реализация его сама не выставляет.

## 12. Как интерпретируются ответы пользователя

Для payroll-triggered intake сервер использует такую модель:

- `monthly_income` = сумма денег именно по текущей выплате;
- `monthly_expenses` = актуальная оценка всех расходов пользователя за целевой месяц на момент опроса;
- `monthly_credit_payments` = актуальная оценка всех кредитных платежей пользователя за целевой месяц на момент опроса;
- `liquid_savings` = доступные накопления на момент опроса.

Если у пользователя в одном месяце несколько выплат, сервер:

- сохраняет отдельный `finance_submission` на каждую выплату;
- суммирует все `monthly_income` за месяц;
- берет `monthly_expenses`, `monthly_credit_payments` и `liquid_savings` из самой свежей submission за месяц;
- после этого пересчитывает `member_finance_snapshots` и `family_dashboard_snapshots`.

## 13. Связка callback с `llm_collection_requests`

Если intake callback приходит с `request_id`, сервер:

- ищет `llm_collection_request` по этому correlation id;
- принимает callback только если запись существует и находится в статусе `accepted`;
- связывает `finance_submission` с найденным `llm_collection_request`;
- после успешного пересчета snapshot-слоя переводит запрос в `completed`.

Если `request_id` не передан, intake обрабатывается как standalone payload без связи с `llm_collection_request`.

Если `request_id` передан, но запись:

- не найдена;
- находится в `pending`;
- находится в `failed`;
- находится в `completed`;
- находится в `cancelled`;

сервер возвращает `422 VALIDATION_ERROR` и не создает производные данные.

## 14. Пример сценария

У пользователя два правила:

- `16 число`;
- `last_day_of_month`.

Для марта 2026:

- первая зарплата: `2026-03-16`, это понедельник, перенос не нужен;
- дата опроса: `2026-03-17`;
- вторая зарплата: `2026-03-31`, это вторник, перенос не нужен;
- дата опроса: `2026-04-01`.

Сервер должен создать два независимых запуска опроса.
Если пользователь ответил на оба опроса, итоговый `monthly_income` за март
должен быть равен сумме обеих выплат.

## 15. Ограничения текущего MVP

В текущем MVP пока не учитываются:

- официальные производственные календари;
- ручные переносы работодателем;
- исключения по конкретным месяцам;
- разовые премии и нерегулярные выплаты;
- несколько стран и разные национальные календари выходных.
