# Логика сервера для запуска опросов после зарплаты

## 1. Назначение

Этот документ описывает серверную логику, по которой создаются задачи на опрос пользователя после получения зарплаты.

Документ не описывает API и БД полностью. Он фиксирует именно правила планировщика и расчета дат.

## 2. Основная идея

Для каждого члена семьи сервер хранит одно или несколько правил зарплатных выплат.

Примеры:

- у одного человека зарплата приходит 16 числа и в последний день месяца;
- у другого человека зарплата приходит 5 и 25 числа.

Для каждого такого события сервер должен:

- вычислить фактическую дату выплаты;
- дождаться нужного дня после выплаты;
- инициировать запуск LLM-агента в `n8n`;
- не допускать повторного запуска для того же зарплатного события.

Каждый такой запуск опроса соответствует одной конкретной выплате, а не целому месяцу целиком.

## 3. Входные данные для планировщика

Планировщик опирается на таблицу:

- `member_payroll_schedules`

Каждая запись в ней описывает одно правило выплаты:

- фиксированный день месяца;
- или последний день месяца;

Правило переноса для MVP общее для всей системы:

- если дата выплаты выпадает на выходной, она переносится на предыдущую пятницу.

## 4. Расчет даты выплаты

### 4.1. Номинальная дата выплаты

Сначала сервер вычисляет номинальную дату выплаты.

Правила:

- если `schedule_type = fixed_day_of_month`, берется соответствующий день месяца;
  если в месяце такого дня нет, берется последний календарный день этого месяца;
- если `schedule_type = last_day_of_month`, берется последний календарный день месяца.

Примеры:

- `16` -> `2026-03-16`
- `31` для апреля 2026 -> `2026-04-30`
- `last_day_of_month` для марта -> `2026-03-31`
- `last_day_of_month` для февраля 2026 -> `2026-02-28`

### 4.2. Фактическая дата выплаты

Затем сервер проверяет, является ли номинальная дата выходным днем.

Для MVP предлагается такое упрощение:

- выходные: суббота и воскресенье;
- официальные праздники пока не учитываются.

Если дата попадает на выходной, применяется общее правило MVP:

- дата всегда переносится на предыдущую пятницу

### 4.3. Примеры переноса

Примеры переноса:

- если дата выпала на субботу, перенос на пятницу;
- если дата выпала на воскресенье, перенос на пятницу.

Другие варианты переноса в MVP не используются.

## 5. Расчет даты запуска опроса

После определения `effective_payroll_date` сервер рассчитывает дату запуска:

- `scheduled_trigger_date = effective_payroll_date + trigger_delay_days`

Если `trigger_delay_days = 1`, опрос стартует на следующий день после фактической выплаты.

Пример:

- фактическая выплата: `2026-03-16`
- `trigger_delay_days = 1`
- запуск опроса: `2026-03-17`

## 6. Работа планировщика

Рекомендуемая схема:

1. Один раз в день сервер запускает джобу, например утром.
2. Джоба выбирает все активные `member_payroll_schedules`.
3. Для каждого правила вычисляется `effective_payroll_date` и `scheduled_trigger_date`.
4. Если `scheduled_trigger_date` совпадает с текущей датой в таймзоне семьи, сервер создает задачу опроса.
5. Сервер проверяет, что для этого зарплатного события еще нет записи в `llm_collection_requests`.
6. Если записи нет, сервер вызывает `n8n`.

## 7. Защита от дублей

Чтобы один и тот же опрос не запускался несколько раз, для каждого зарплатного события нужна уникальность.

Для MVP достаточно считать уникальным сочетание:

- `payroll_schedule_id`
- `effective_payroll_date`

Это означает:

- для одной и той же зарплаты опрос можно запустить только один раз.

## 8. Какой месяц считать периодом сбора

Для MVP рекомендуется простое правило:

- `period_month` и `period_year` совпадают с месяцем и годом фактической даты выплаты.

Пример:

- зарплата пришла `2026-03-31`
- опрос стартовал `2026-04-01`
- данные относятся к периоду `2026-03`

Это особенно важно для выплат в конце месяца.

## 8.1. Как интерпретировать ответы пользователя

Для payroll-triggered intake сервер использует такую модель:

- `monthly_income` = сумма денег именно по текущей выплате;
- `monthly_expenses` = актуальная оценка всех расходов пользователя за целевой месяц на момент опроса;
- `monthly_credit_payments` = актуальная оценка всех кредитных платежей пользователя за целевой месяц на момент опроса;
- `liquid_savings` = доступные накопления на момент опроса.

Если у пользователя в одном месяце несколько выплат, сервер:

- сохраняет отдельный `finance_submission` на каждую выплату;
- суммирует все `monthly_income` за месяц;
- берет `monthly_expenses`, `monthly_credit_payments` и `liquid_savings` из самой свежей submission за месяц;
- после этого пересчитывает `member_finance_snapshots` и `family_dashboard_snapshots`.

## 9. Пример сценария

У пользователя два правила:

- `16 число`
- `last_day_of_month`

Для марта 2026:

- первая зарплата: `2026-03-16`, это понедельник, перенос не нужен
- дата опроса: `2026-03-17`
- вторая зарплата: `2026-03-31`, это вторник, перенос не нужен
- дата опроса: `2026-04-01`

Сервер должен создать два независимых запуска опроса.
Если пользователь ответил на оба опроса, итоговый `monthly_income` за март должен быть равен сумме обеих выплат.

## 10. Ограничения MVP

В MVP пока не учитываются:

- официальные производственные календари;
- ручные переносы работодателем;
- исключения по конкретным месяцам;
- разовые премии и нерегулярные выплаты;
- несколько стран и разные национальные календари выходных.

## 11. Что можно добавить позже

После MVP можно расширить логику:

- таблицей праздничных дней;
- таблицей ручных исключений по зарплатным датам;
- разными типами выплат: зарплата, аванс, премия;
- повторными напоминаниями, если пользователь не ответил;
- отдельным статусом завершенности опроса по каждому зарплатному событию.
