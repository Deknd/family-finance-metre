# Задача: выстроить понятное и безопасное логирование сервера

## Вступление

Цель этой задачи - выстроить такой подход к логированию, при котором логи помогают быстро понять, что произошло в системе, на каком этапе возникла проблема и как найти ее источник. При этом логи не должны превращаться в хранилище персональных данных, секретов или полных пользовательских payload.

Задача сформулирована не как список конкретных точек в коде, а как набор правил и ориентиров. Идея в том, чтобы по этим правилам можно было самостоятельно находить места для логирования, принимать осознанные решения и постепенно вырабатывать инженерную привычку писать полезные логи.

## Контекст проекта

У сервера есть несколько разных потоков, для которых логирование особенно важно:

- `device -> server` - входящие запросы от физического устройства;
- `server -> n8n` - исходящий вызов во внешний интеграционный контур;
- `n8n -> server` - входящий callback с результатами опроса;
- scheduler-задачи, которые запускают server-side процессы по времени;
- внутренние пересчеты snapshot-данных и переходы состояний интеграционных сущностей.

Для интеграционной цепочки в проекте уже есть важный технический идентификатор: `request_id`. Он должен рассматриваться как основной correlation id цепочки `server -> n8n -> server`. Для веток, где `request_id` отсутствует, нужен отдельный `correlation_id`, который создается на границе обработки и проходит через все связанные логи.

## Цель задачи

Нужно сформировать единый стандарт логирования уровня `production base` для серверной части проекта.

Под этим уровнем понимается:

- структурированное логирование;
- понятные и предсказуемые уровни логов;
- сквозная корреляция событий;
- минимально достаточный набор технических полей;
- отсутствие персональных данных, секретов и финансовых значений в логах.

## Почему это важно

Хороший лог нужен не для количества записей и не для ощущения "что-то пишется". Он нужен для того, чтобы:

- восстановить цепочку событий по одному запросу или одной джобе;
- быстро понять, где именно произошел отказ;
- отличить штатное отклонение от реального сбоя;
- сократить время на диагностику без просмотра сырых payload и пользовательских данных.

Если из лога нельзя понять проблему без открытия базы, request body или внешней системы, значит логирование организовано слабо. Если из логов можно узнать личные данные пользователя, значит оно организовано небезопасно.

## 1. Ожидаемый результат

После выполнения задачи логирование должно соответствовать следующим принципам:

- логи структурированы и читаются как последовательность технических событий;
- у логов есть единый и предсказуемый набор полей;
- по `request_id` или `correlation_id` можно собрать всю цепочку обработки;
- у ключевых событий есть понятные `INFO`, у значимых отклонений есть `WARN`, у неожиданных сбоев есть `ERROR`;
- одна и та же проблема не логируется полноценно несколько раз на каждом слое;
- лог не содержит PII, секретов, финансовых сумм и сырых тел запросов;
- по одному логу или небольшой связке логов можно понять, что произошло, без чтения полного payload.

## 2. Технологии и подход

Для этой задачи нужно использовать штатный стек логирования Spring Boot:

- `SLF4J` как единый API для вызова логов;
- `Logback` как базовую реализацию;
- уровни логирования `ERROR`, `WARN`, `INFO`, `DEBUG`;
- `MDC` для передачи correlation-контекста между связанными логами;
- структурированный формат логов для production;
- человекочитаемый pattern-формат для local-профиля, если это удобно для разработки.

Практический смысл технологий:

- `SLF4J` позволяет не привязываться к конкретной реализации;
- `Logback` хорошо интегрирован со Spring Boot и покрывает типовые потребности проекта;
- `MDC` помогает автоматически добавлять корреляционные поля в каждую запись лога в рамках одной ветки обработки;
- JSON-формат удобен для production, потому что его проще фильтровать, искать и агрегировать;
- локальный читаемый pattern полезен для ручной отладки, если он сохраняет те же смысловые поля.

Следующий уровень зрелости может включать централизованный сбор логов, поиск по ним и связку с трассировкой. Но для этой задачи это не обязательная часть. Сначала нужен хороший локальный стандарт самих логов.

## 3. Правила поиска мест для логирования

Лог нужно добавить там, где происходит технически или бизнес-значимое событие.

### Лог нужно добавить там, где:

- начинается обработка на системной границе;
- успешно завершается обработка на системной границе;
- приходит отказ аутентификации или авторизации;
- происходит валидационное отклонение входных данных;
- меняется жизненный цикл сущности или интеграционного процесса;
- вызывается внешняя интеграция;
- завершается scheduler-джоба или batch-обработка;
- нарушается ожидаемый инвариант;
- возникает неожиданный сбой, который прерывает поток.

### Что это означает на практике

- На границе входящего HTTP-запроса лог нужен, когда сервер принимает запрос и когда возвращает итоговый результат.
- На границе исходящего вызова лог нужен перед началом обращения во внешнюю систему и после получения результата.
- На lifecycle-переходе лог нужен, когда состояние сущности меняется так, что это влияет на дальнейшее поведение системы.
- В scheduler-потоке лог нужен не на каждом мельчайшем шаге, а на уровне полезного итога: что было обработано, что было создано, что было пропущено, что завершилось ошибкой.
- На ошибке лог нужен в той точке, где уже понятен итоговый смысл проблемы и собран технический контекст.

### Где лог обычно не нужен

- при простом чтении из репозитория без изменения состояния;
- при тривиальных внутренних вызовах, которые не добавляют нового контекста;
- там, где ошибка уже была полноценно залогирована выше или ниже по стеку;
- там, где лог не помогает диагностике и только увеличивает шум.

Главный принцип: логировать не каждый шаг подряд, а только те места, которые меняют понимание происходящего.

## 4. Политика данных в логах

Логи должны содержать только безопасный технический контекст.

### Можно логировать

- `request_id`;
- `correlation_id`;
- `submission_id`;
- `family_id`;
- `member_id`;
- `device_id`;
- `workflow_run_id`;
- имя потока или сценария обработки;
- endpoint или путь запроса;
- HTTP-метод;
- HTTP-статус;
- код ошибки;
- класс исключения;
- длительность операции;
- счетчики обработанных, созданных, пропущенных и ошибочных элементов;
- технический результат операции: `accepted`, `completed`, `failed`, `skipped`, `validation_error` и подобные безопасные значения.

### Нельзя логировать

- `X-Device-Token`;
- `X-API-Key`;
- bearer token;
- hash токена устройства;
- `telegram_chat_id`;
- `telegram_username`;
- имя, фамилию, display name пользователя;
- название семьи;
- `notes`;
- полные request body и response body;
- raw JSON payload;
- любые финансовые суммы;
- любые фрагменты исключений или текстов, в которые могли попасть пользовательские данные или секреты.

Если проблему трудно понять без payload, это не повод писать payload в лог. Вместо этого нужно выделить безопасную техническую причину ошибки и логировать ее в нормализованном виде.

## 5. Правила выбора уровня логирования

- `ERROR` - неожиданные ошибки, падения интеграции, нарушения инвариантов, сбои, из-за которых поток не может корректно завершиться.
- `WARN` - отказ безопасности, ожидаемое, но важное отклонение, бизнес-отказ или контрактная проблема, которая не является нормальным успешным сценарием.
- `INFO` - ключевые успешные события, переходы состояний, начало и завершение значимого сценария, полезные summary-логи.
- `DEBUG` - временная или углубленная диагностика, которую можно включать точечно; не должен быть основным источником понимания production-проблем.

При выборе уровня полезно задавать себе вопрос: если эта запись исчезнет из production-логов, станет ли заметно сложнее понимать поведение системы? Если нет, возможно, это не `INFO`.

## 6. Минимальный набор полей каждого полезного лога

Каждый полезный лог должен по возможности содержать минимальный, но достаточный набор унифицированных полей:

- `event`;
- `flow`;
- `result`;
- `correlation_id` и/или `request_id`;
- `http_status`, если лог относится к HTTP-обработке;
- `duration_ms`, если речь идет об операции с заметной длительностью;
- `error_code`, если есть нормализованная кодовая причина;
- `exception_class`, если лог связан с исключением.

Дополнительно можно включать безопасные идентификаторы и счетчики, если они помогают понять ситуацию.

Важно не просто добавлять эти поля, а называть их единообразно во всем проекте. Польза от структурированного логирования сильно падает, если одна и та же сущность в разных местах называется по-разному.

## 7. Пошаговый план выполнения

### Шаг 1. Провести инвентаризацию потоков

Нужно выписать все основные ветки обработки, в которых есть смысл видеть логическую цепочку:

- входящие HTTP-запросы;
- исходящие интеграционные вызовы;
- scheduler-джобы;
- внутренние переходы состояний;
- пересчеты производных данных.

На что подумать перед реализацией:
какие события действительно важны для диагностики, а какие создадут только шум?

### Шаг 2. Утвердить безопасные поля логирования

Нужно заранее определить, какие технические поля разрешены к логированию, а какие запрещены. Это лучше сделать до массового добавления логов, чтобы потом не исправлять десятки небезопасных записей.

На что подумать перед реализацией:
можно ли понять проблему по техническим идентификаторам и статусам без вывода пользовательских данных?

### Шаг 3. Ввести корреляцию через `MDC`

Нужно обеспечить, чтобы у связанных логов автоматически появлялся общий корреляционный контекст:

- использовать `request_id`, если он уже существует как часть потока;
- генерировать `correlation_id`, если поток начинается без `request_id`.

На что подумать перед реализацией:
где начинается жизнь correlation id, где он должен очищаться и как убедиться, что один запрос не загрязняет другой?

### Шаг 4. Добавить логи на системных границах

Нужно покрыть логами начало и завершение основных системных границ:

- входящий запрос;
- исходящий интеграционный вызов;
- scheduler-запуск.

На что подумать перед реализацией:
хватит ли одного стартового и одного итогового лога, чтобы понять судьбу операции?

### Шаг 5. Добавить логи на lifecycle-событиях

Нужно зафиксировать те изменения статуса или состояния, которые важны для понимания потока:

- создание;
- принятие;
- завершение;
- отказ;
- пропуск как уже обработанного сценария.

На что подумать перед реализацией:
изменяет ли этот переход последующее поведение системы настолько, чтобы его стоило видеть в логах?

### Шаг 6. Добавить логи на интеграционных ошибках и отклонениях

Нужно логировать не только факт ошибки, но и ее технический контекст:

- на каком этапе произошел сбой;
- каков был технический результат;
- есть ли correlation id;
- относится ли это к контрактной ошибке, сетевому сбою, отказу безопасности или бизнес-валидации.

На что подумать перед реализацией:
можно ли сформулировать проблему кратко и безопасно, не выводя payload и пользовательские поля?

### Шаг 7. Привести конфиг логирования к единому формату

Нужно сделать так, чтобы в production был структурированный формат, а в local при необходимости - читаемый формат с теми же ключевыми полями.

На что подумать перед реализацией:
останутся ли ключевые поля заметными и одинаковыми независимо от окружения?

### Шаг 8. Проверить отсутствие PII, секретов и дублирования

Нужно провести финальную проверку:

- нет ли в логах чувствительных полей;
- нет ли одинаковых `ERROR` на одну и ту же проблему;
- можно ли понять путь обработки по correlation-полям;
- не перегружены ли обычные успешные сценарии лишним шумом.

На что подумать перед реализацией:
если открыть production-лог во время инцидента, получится ли быстро найти причину без ручной фильтрации сотен однотипных записей?

## 8. Примеры хороших логов

Ниже приведены безопасные шаблоны логов. Это именно образцы структуры и смысла, а не привязка к конкретным классам.

```text
level=INFO event=http_request_received flow=device_to_server result=started correlation_id=<correlation_id> path=/api/v1/device/dashboard http_method=GET
```

```text
level=INFO event=integration_call_completed flow=server_to_n8n result=accepted request_id=<request_id> workflow_run_id=<workflow_run_id> http_status=202 duration_ms=<duration_ms>
```

```text
level=WARN event=validation_rejected flow=n8n_to_server result=validation_error correlation_id=<correlation_id> error_code=VALIDATION_ERROR http_status=422
```

```text
level=ERROR event=integration_call_failed flow=server_to_n8n result=failed request_id=<request_id> error_code=INTEGRATION_FAILURE exception_class=<exception_class> duration_ms=<duration_ms>
```

```text
level=INFO event=scheduler_run_finished flow=payroll_collection_scheduler result=completed correlation_id=<correlation_id> processed_count=<processed_count> created_count=<created_count> skipped_count=<skipped_count> error_count=<error_count> duration_ms=<duration_ms>
```

## 9. Критерии приемки

Задача считается выполненной, если выполняются все условия:

- любую связанную цепочку можно собрать по `request_id` или `correlation_id`;
- по логу интеграционной ошибки видно, на каком этапе произошел сбой и какова техническая причина;
- в логах нет персональных данных, секретов и финансовых значений;
- на одну проблему не появляется каскад одинаковых полноценных `ERROR`;
- по логам можно понять проблему без просмотра пользовательского payload;
- формат и имена основных полей единообразны по всему проекту.

## 10. Вопросы для самопроверки

Перед добавлением каждого нового лога стоит задавать себе несколько вопросов:

- Это действительно граница системы или важное изменение состояния, а не внутренний шум?
- Нужен ли здесь `INFO`, или запись будет только засорять обычный сценарий?
- Можно ли понять проблему без payload?
- Не попадает ли в лог персональная информация, секрет или финансовое значение?
- Есть ли у этой ветки `request_id` или `correlation_id`?
- Не логирую ли я одну и ту же ошибку второй раз без нового контекста?
- Поможет ли эта запись быстрее найти причину инцидента?
- Если оставить только этот лог и соседние записи по той же корреляции, станет ли цепочка понятной?

## Итоговая формулировка задачи

Нужно не просто "добавить логи", а выстроить понятную и безопасную систему наблюдаемости базового уровня. Результатом должен стать такой стандарт, по которому разработчик умеет сам находить правильные точки логирования, отличать полезный лог от шума и удерживать границу между технической диагностикой и недопустимым выводом пользовательских данных.
