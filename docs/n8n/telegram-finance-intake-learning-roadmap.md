# Учебный roadmap реализации `n8n` intake-агента

## 1. Назначение

Этот документ нужен как практический список задач, который можно брать в работу по шагам и самостоятельно реализовывать в `n8n`.

Цель roadmap:

- провести реализацию от самого простого mock webhook до полного MVP;
- не перепутать учебные этапы с реальной границей MVP;
- на каждом этапе понимать, что именно нужно изучить в `n8n`;
- иметь четкие критерии готовности, а не только общий план.

Документ опирается на:

- [llm-agent-trigger-spec.md](./llm-agent-trigger-spec.md)
- [telegram-finance-intake-agent-spec.md](./telegram-finance-intake-agent-spec.md)
- [telegram-finance-intake-resilience-algorithm.md](./telegram-finance-intake-resilience-algorithm.md)
- [api-spec.md](../server/api-spec.md)

## 2. Как читать этот документ

Логика такая:

- задачи из блоков `0-2` нужны для обучения и набора рабочего контура;
- задачи из блока `3` закрывают именно MVP;
- задачи из блока `4` идут только после MVP и относятся к production-hardening;
- если какой-то учебный шаг реализован упрощенно, это нормально, но к моменту закрытия MVP все временные упрощения должны быть убраны.

## 3. Граница MVP

MVP считается завершенным не после mock-решения и не после детерминированного Telegram-прототипа.

MVP считается завершенным только когда одновременно выполнены все условия:

- реализованы оба workflow из целевой спецификации;
- стартовый webhook возвращает `202 Accepted` только после успешной инициализации intake-сессии;
- сессия хранится в `Redis`;
- диалог идет через `Telegram Trigger`;
- подключены `AI Agent` и `Structured Output Parser`;
- callback на сервер соответствует фактическому API-контракту;
- поддержаны `review`, подтверждение перед submit, `cancel`, `restart`, `help`, timeout и идемпотентность;
- пройден полный сценарий `server -> n8n -> Telegram -> server`.

## 4. Порядок выполнения

Рекомендуемый порядок:

1. блок `0` — подготовка среды и контрактов;
2. блок `1` — учебный mock-контур;
3. блок `2` — stateful Telegram-прототип без полного AI-MVP;
4. блок `3` — доведение до spec-complete MVP;
5. блок `4` — production-hardening.

## 5. Блок 0. Подготовка среды и контрактов

### Задача 00. Подтвердить рабочую версию `n8n` и набор узлов

Что сделать:

- выбрать конкретную версию `n8n` для локальной разработки;
- проверить наличие узлов `Webhook`, `Respond to Webhook`, `Telegram Trigger`, `Telegram`, `AI Agent`, `Structured Output Parser`, `HTTP Request`, `Redis`, `Redis Chat Memory`;
- зафиксировать решение в отдельной заметке или README локального контура.

Что изучаете:

- как версия `n8n` влияет на доступность узлов и параметры workflow.

Критерий готовности:

- вы точно знаете, на какой версии строите обучение и MVP;
- у вас нет скрытого риска, что нужного узла нет в инстансе.

### Задача 01. Поднять локальный self-hosted контур `n8n + Redis`

Что сделать:

- поднять локальный `n8n`;
- поднять локальный `Redis`;
- проверить, что `n8n` доступен из браузера, а `Redis` доступен для credential в `n8n`.

Что изучаете:

- базовую self-hosted установку `n8n`;
- работу с credentials и внешними зависимостями.

Критерий готовности:

- локальный инстанс запускается стабильно;
- можно создать credential для `Redis`.

### Задача 02. Подготовить матрицу secrets и environment variables

Что сделать:

- завести значения для webhook token, callback API key, server callback URL, Telegram bot token, TTL сессии;
- договориться, какие секреты живут в `.env`, а какие только в credentials `n8n`;
- отдельно зафиксировать переменные для `dev`.

Что изучаете:

- разницу между env-переменными и credentials;
- как не хардкодить секреты в workflow.

Критерий готовности:

- все нужные секреты разложены по понятной схеме;
- ни один workflow не требует хардкода токена в узле.

### Задача 03. Зафиксировать контрактные артефакты для реализации

Что сделать:

- выписать стартовый webhook request/response контракт;
- выписать callback-контракт в сервер;
- выписать структуру business session state и ключи `Redis`;
- выписать схему structured output для AI-агента.

Что изучаете:

- как строить workflow от контракта, а не от импровизации в UI.

Критерий готовности:

- перед началом сборки workflow у вас есть 4 отдельных артефакта:
  стартовый контракт, callback-контракт, session state, structured output schema.

## 6. Блок 1. Учебный mock-контур

### Задача 04. Собрать самый простой start webhook

Что сделать:

- создать workflow с `Webhook`;
- принимать `POST /webhook/finance-intake-start`;
- всегда возвращать `202 Accepted` c `status`, `request_id`, `workflow_run_id`.

Что изучаете:

- trigger workflow;
- expressions;
- `Respond to Webhook`.

Критерий готовности:

- локальный `curl` или Postman-запрос получает корректный `202` и JSON-ответ.

### Задача 05. Добавить аутентификацию и валидацию mock webhook

Что сделать:

- проверить `Authorization: Bearer`;
- провалидировать минимально обязательные поля;
- вернуть ветки `401` и `422`.

Что изучаете:

- guards;
- early-return ветки;
- базовую обработку ошибок в webhook workflow.

Критерий готовности:

- у start workflow есть 3 устойчивых ветки: `202`, `401`, `422`.

### Задача 06. Добавить mock-идемпотентность start webhook

Что сделать:

- добавить ветку `409 Conflict` для duplicate `request_id`;
- пока допускается mock-логика без настоящего `Redis`, если это нужно только для учебного шага.

Что изучаете:

- как моделировать duplicate-ветку до появления полноценного session state.

Критерий готовности:

- start workflow может показать все 4 базовые статуса: `202`, `401`, `409`, `422`.

### Задача 07. Собрать асинхронный mock полного цикла

Что сделать:

- после успешного `202` запускать mock callback;
- сначала отправлять callback в mock endpoint;
- затем переключить callback на реальный серверный endpoint.

Что изучаете:

- разницу между быстрым HTTP-ответом и продолжением работы workflow;
- асинхронную цепочку `start -> callback`.

Критерий готовности:

- webhook отвечает сразу;
- callback приходит отдельно и не блокирует start-запрос.

## 7. Блок 2. Stateful Telegram-прототип

### Задача 08. Реализовать модель session state в `Redis`

Что сделать:

- создать схему ключей:
  `finance:intake:session:{request_id}`,
  `finance:intake:chat:{telegram_chat_id}`,
  `finance:intake:dedupe:telegram-update:{update_id}`;
- определить TTL;
- хранить session state отдельно от chat memory.

Что изучаете:

- stateful orchestration в `n8n`;
- разделение бизнес-состояния и conversational memory.

Критерий готовности:

- после старта intake вы можете увидеть в `Redis` понятную запись сессии и chat index.

### Задача 09. Собрать `Workflow A` в детерминированном виде

Что сделать:

- принять стартовый webhook;
- проверить payload;
- определить canonical ordering полей;
- создать session state;
- отправить первое сообщение в Telegram;
- вернуть `202` только после успешной инициализации.

Что изучаете:

- orchestration workflow;
- границу accepted;
- связку `Webhook -> Redis -> Telegram -> Respond to Webhook`.

Критерий готовности:

- после start webhook пользователь получает первое сообщение, а сессия уже лежит в `Redis`.

### Задача 10. Собрать `Workflow B` без AI, но с полной state machine по шагам

Что сделать:

- использовать `Telegram Trigger`;
- читать active-chat index и session state;
- задавать вопросы в fixed canonical order;
- последовательно собирать 4 поля;
- показывать summary и ждать подтверждения.

Что изучаете:

- event-driven Telegram workflow;
- хранение состояния между сообщениями;
- ветвление по статусам сессии.

Критерий готовности:

- пользователь может вручную пройти весь сценарий до `review` без AI.

### Задача 11. Добавить команды пользователя

Что сделать:

- реализовать `отмена`;
- реализовать `начать заново`;
- реализовать `помощь`;
- описать, какие поля session state при этом меняются.

Что изучаете:

- service commands в event-driven workflow;
- аккуратный reset состояния.

Критерий готовности:

- команды не ломают сессию и предсказуемо меняют ее статус.

### Задача 12. Добавить базовую устойчивость Telegram-прототипа

Что сделать:

- добавить дедупликацию `telegram_update_id`;
- не допускать две активные сессии на один `telegram_chat_id`;
- обработать stale chat-index;
- зафиксировать политику TTL.

Что изучаете:

- почему без дедупликации и chat-index stateful workflow быстро становится неуправляемым.

Критерий готовности:

- duplicate update не ломает сценарий;
- новый старт на занятый чат предсказуемо отклоняется.

## 8. Блок 3. Доведение до MVP по спецификации

### Задача 13. Подключить `AI Agent` и `Structured Output Parser`

Что сделать:

- подключить `AI Agent`;
- задать ему строгий prompt только на интерпретацию текущего ответа;
- подключить `Structured Output Parser` по заранее согласованной схеме;
- не давать AI управлять lifecycle-решениями.

Что изучаете:

- как встраивать LLM в workflow безопасно и контролируемо.

Критерий готовности:

- AI возвращает structured output, а не свободный текст;
- output можно валидировать deterministic-веткой.

### Задача 14. Реализовать deterministic-валидацию результата AI

Что сделать:

- принимать только допустимые поля в `field_updates`;
- принимать только числа `>= 0`;
- поддержать clarification и recovery policy;
- переводить в `needs_human_review` или `failed`, если AI не смог надежно извлечь значение.

Что изучаете:

- как ограничивать LLM deterministic-логикой.

Критерий готовности:

- AI не может самовольно завершить сессию или отправить callback;
- плохой output не разрушает workflow.

### Задача 15. Реализовать `review` и явное подтверждение перед submit

Что сделать:

- после сбора всех полей переводить сессию в `review`;
- формировать summary в canonical order;
- разрешать либо подтверждение, либо возврат к редактированию конкретного поля.

Что изучаете:

- как строить контролируемую человеческую проверку перед внешним submit.

Критерий готовности:

- callback не уходит без явного подтверждения пользователя.

### Задача 16. Собрать итоговый callback payload по фактическому серверному DTO

Что сделать:

- формировать payload только из normalized session state;
- вернуть `request_id` без изменений;
- использовать один `external_submission_id` на всю сессию;
- правильно заполнить `meta.telegram_chat_id`, `meta.confidence`, `meta.notes`.

Что изучаете:

- mapping внутреннего состояния workflow в внешний API-контракт.

Критерий готовности:

- итоговый JSON проходит по требованиям [api-spec.md](../server/api-spec.md).

### Задача 17. Реализовать callback handling и terminal states

Что сделать:

- обработать `202 Accepted`;
- обработать `409 DUPLICATE_SUBMISSION` как успешную дедупликацию;
- обработать `401`, `422`, сетевые ошибки и локальный `failed`;
- корректно удалять active chat index при терминальных статусах.

Что изучаете:

- как замыкать интеграционную цепочку и не получать зомби-сессии.

Критерий готовности:

- после submit сессия всегда приходит в один из понятных terminal states.

### Задача 18. Реализовать retry, timeout и smoke-check полного сценария

Что сделать:

- добавить retry policy callback;
- добавить timeout policy и закрытие заброшенных сессий;
- проверить сценарии happy path, duplicate, validation error, cancel, restart, timeout, needs_human_review;
- пройти полный E2E сценарий `server -> n8n -> Telegram -> server`.

Что изучаете:

- эксплуатационную устойчивость workflow;
- проверку edge cases до выхода в MVP.

Критерий готовности:

- после выполнения этой задачи MVP считается завершенным.

## 9. Блок 4. Production-hardening после MVP

### Задача 19. Разнести окружения и секреты

Что сделать:

- разделить `dev`, `test`, `prod` credentials;
- описать, какие значения меняются между окружениями;
- убрать все учебные и временные значения.

Критерий готовности:

- production workflow не зависит от локальных моков и тестовых токенов.

### Задача 20. Добавить observability и runbook

Что сделать:

- определить обязательные лог-события;
- завести минимальные алерты по `failed` и `needs_human_review`;
- описать действия при сбоях `Redis`, `Telegram`, server callback.

Критерий готовности:

- у вас есть понятный operational playbook на типовые сбои.

### Задача 21. Подготовить экспорт, backup и восстановление

Что сделать:

- описать, как экспортируются workflow;
- описать, как хранится версия workflow в Git;
- описать, как восстановить контур после потери инстанса.

Критерий готовности:

- есть минимально безопасная процедура восстановления.

### Задача 22. Провести production readiness review

Что сделать:

- пройтись по всем инвариантам из resilience-алгоритма;
- проверить, что временные упрощения учебных этапов убраны;
- отдельно проверить security, duplicate handling и terminal-state policy.

Критерий готовности:

- решение можно считать production-ready для первой поставки.

## 11. Короткая рекомендация по стилю работы

Чтобы обучение шло быстрее, лучше придерживаться такого ритма:

- сначала собирать минимально рабочую ветку поведения;
- потом сразу фиксировать, что именно вы упростили;
- только после этого усложнять следующий шаг;
- после каждой крупной задачи делать короткий smoke-check и обновлять заметки по архитектуре.

Тогда вы не потеряете понимание, зачем в workflow появился каждый новый узел и каждая новая ветка.
