# План реализации двухуровневого n8n AI intake-агента

## 1. Назначение

Этот документ описывает пошаговый план реализации двухуровневого `n8n`-решения для сбора финансовых данных через Telegram.

План построен так, чтобы:

- разбить решение на небольшие реализуемые задачи;
- сначала собрать устойчивую архитектурную основу;
- потом замкнуть живой AI-диалог и callback на сервер;
- в конце получить проверяемый сценарий
  `server -> n8n -> Telegram -> server`.

Документ опирается на:

- [telegram-finance-intake-agent-spec.md](./telegram-finance-intake-agent-spec.md)
- [llm-agent-trigger-spec.md](./llm-agent-trigger-spec.md)
- [api-spec.md](../server/api-spec.md)

## 2. Что считаем готовым результатом

Решение можно считать реализованным, когда одновременно выполняются такие условия:

- сервер вызывает `Webhook` orchestration workflow;
- orchestration workflow сразу отвечает `202 Accepted`;
- в `n8n` создается intake-сессия с сохранением state в Redis;
- пользователю в Telegram приходит стартовое сообщение;
- дальнейший диалог идет через `Telegram Trigger`;
- conversational AI-агент последовательно собирает все 4 поля;
- итоговый payload формируется строго по серверному DTO;
- callback успешно уходит на `POST /api/v1/intake/user-finance-data`;
- при retry повторно используется тот же `external_submission_id`;
- ошибки, cancel, timeout и escalation обрабатываются предсказуемо.

## 3. Границы первой реализации

В первую реализацию входят:

- только `Telegram`;
- два workflow в `n8n`;
- `Redis` для session state;
- `Redis Chat Memory` для chat memory;
- structured output для AI-агента;
- callback в серверное API;
- базовый manual review / escalation path.

В первую реализацию не входят:

- voice input;
- файлы и фото;
- multi-channel routing;
- полноценная операторская панель;
- хранение аналитики разговоров в отдельной БД;
- community HITL-ноды как обязательная часть решения.

## 4. Рекомендуемый порядок реализации

Рекомендуется идти блоками:

1. инфраструктура и контракты;
2. `Workflow A / Intake Orchestrator Agent`;
3. `Workflow B / Telegram Conversation Agent`;
4. callback на сервер и идемпотентность;
5. fallback, cancel, timeout, review;
6. тестирование и smoke-check.

## 5. План задач

## Блок A. Подготовить основу интеграции

### Задача 01. Подтвердить целевую версию и доступные узлы `n8n`

Что сделать:

- сверить фактическую версию `n8n` с версией, совместимой с MCP-документацией;
- подтвердить наличие узлов:
  `Webhook`, `Respond to Webhook`, `Telegram Trigger`, `Telegram`,
  `AI Agent`, `Structured Output Parser`, `HTTP Request`, `Redis`,
  `Redis Chat Memory`, `Execute Sub-workflow` или `Call n8n Sub-Workflow Tool`;
- зафиксировать, какие именно credentials будут использоваться.

Результат:

- реализация идет на подтвержденном наборе узлов без скрытых несовместимостей.

### Задача 02. Подготовить credentials и environment для `n8n`

Что сделать:

- создать credential для Telegram bot;
- создать credential для Redis;
- создать credential для LLM-провайдера;
- создать credential для callback HTTP-запросов на сервер;
- завести переменные окружения для URL сервера, API key, shared secret webhook-вызова от сервера, TTL сессии.

Результат:

- workflow можно собирать без хардкода секретов.

### Задача 03. Зафиксировать схемы входа и выхода

Что сделать:

- перенести в `n8n` фактический контракт стартового webhook из
  `docs/n8n/llm-agent-trigger-spec.md`;
- перенести фактический callback-контракт из
  `docs/server/api-spec.md`;
- отдельно описать внутреннюю JSON-схему structured output для conversational AI-агента;
- отдельно описать JSON-структуру session state в Redis.

Результат:

- у реализации есть стабильные JSON-контракты на всех границах.

## Блок B. Реализовать `Workflow A / Intake Orchestrator Agent`

### Задача 04. Собрать стартовый webhook workflow

Что сделать:

- создать `Webhook` на `POST /webhook/finance-intake-start`;
- принять стартовый payload;
- проверить заголовок аутентификации;
- провалидировать обязательные поля до запуска AI-логики;
- подготовить `workflow_run_id` и correlation metadata.

Результат:

- `n8n` может безопасно принимать стартовую команду от сервера.

### Задача 05. Реализовать orchestration-логику запуска сессии

Что сделать:

- проверить допустимость сценария для Telegram intake;
- определить, существует ли уже активная сессия по `request_id`;
- при дубле вернуть `409 Conflict` либо применить policy reuse, если она будет выбрана позже;
- при отсутствии дубля создать стартовый session state;
- сгенерировать `external_submission_id` один раз на всю intake-сессию.

Результат:

- orchestration workflow становится идемпотентным и предсказуемым.

### Задача 06. Добавить вызов подзадачи или инициализации conversational-сессии

Что сделать:

- вызвать вспомогательный подworkflow либо локальный блок инициализации через
  `Execute Sub-workflow` или `Call n8n Sub-Workflow Tool`;
- сохранить стартовую структуру `business session state` в Redis;
- при необходимости отправить первое сообщение в Telegram.

Результат:

- orchestration workflow реально открывает новую intake-сессию.

### Задача 07. Вернуть корректный `202 Accepted`

Что сделать:

- использовать `Respond to Webhook`;
- вернуть `status`, `request_id`, `workflow_run_id`;
- завершать HTTP-взаимодействие сразу после успешной инициализации сессии;
- отдельно обработать `401`, `409` и `422`.

Результат:

- сервер получает быстрый асинхронный ответ без ожидания диалога с пользователем.

## Блок C. Реализовать `Workflow B / Telegram Conversation Agent`

### Задача 08. Собрать event-driven Telegram workflow

Что сделать:

- создать `Telegram Trigger`;
- извлекать `telegram_chat_id` и текст сообщения;
- по `telegram_chat_id` находить активную intake-сессию;
- завершать обработку понятным сообщением, если сессия не найдена.

Результат:

- каждое пользовательское сообщение корректно маршрутизируется в active session.

### Задача 09. Подключить session state и chat memory

Что сделать:

- загрузить `business session state` из Redis;
- подключить `Redis Chat Memory` для разговорного контекста;
- явно развести бизнес-состояние и chat memory;
- на каждый шаг обновлять `updated_at`, `last_question`, `missing_fields`, `answers`, `notes`.

Результат:

- AI-агент получает устойчивый контекст и не зависит только от истории чата.

### Задача 10. Реализовать prompt и structured output conversational AI-агента

Что сделать:

- собрать system prompt со строгими правилами интерпретации финансовых полей;
- подключить `Structured Output Parser`;
- зафиксировать минимальный внутренний ответ:
  `reply_text`, `field_updates`, `missing_fields`, `next_state`,
  `is_complete`, `confidence`, `notes`, `needs_human_review`, `should_submit`;
- ограничить AI-агента одним основным вопросом за шаг.

Результат:

- агент возвращает не свободный текст, а контролируемую структуру.

### Задача 11. Реализовать state machine сбора полей

Что сделать:

- поддержать состояния:
  `awaiting_income`, `awaiting_expenses`,
  `awaiting_credit_payments`, `awaiting_savings`, `review`,
  `submitting`, `completed`, `failed`, `cancelled`, `needs_human_review`;
- после каждого ответа обновлять состояние;
- поддержать максимум одно уточнение на поле;
- учитывать `allow_approximate_values`.

Результат:

- живой диалог становится детерминированным и проверяемым.

### Задача 12. Реализовать review и подтверждение перед submit

Что сделать:

- после сбора всех полей показать summary пользователю;
- дождаться явного подтверждения;
- только после подтверждения переводить сессию в `submitting`;
- при отказе пользователя возвращать сессию в нужное состояние редактирования.

Результат:

- серверный callback уходит только после подтверждения пользователем.

## Блок D. Замкнуть callback на сервер

### Задача 13. Реализовать сборку итогового payload

Что сделать:

- собрать итоговый JSON из нормализованного session state;
- строго соблюдать ограничения DTO:
  UUID в `family_id` и `member_id`,
  `source = telegram`,
  `collected_at` в ISO-8601,
  `meta.confidence` из допустимого набора;
- переносить `notes` и признаки approximate input в `meta.notes`.

Результат:

- итоговый payload всегда соответствует серверному контракту.

### Задача 14. Реализовать HTTP callback и retry policy

Что сделать:

- отправлять `POST /api/v1/intake/user-finance-data` через `HTTP Request`;
- передавать `X-API-Key`;
- при retry повторно использовать тот же `external_submission_id`;
- различать ответы:
  `202 Accepted`,
  `409 DUPLICATE_SUBMISSION`,
  `422 VALIDATION_ERROR`,
  сетевые ошибки.

Результат:

- callback устойчив к повторам и ошибкам сети.

### Задача 15. Реализовать завершение сессии после callback

Что сделать:

- при `202` переводить session state в `completed`;
- при `409` трактовать результат как успешную дедупликацию и тоже завершать сессию;
- при `422` переводить сессию в `failed` или `needs_human_review` в зависимости от причины;
- сохранять последний server response в техническом логе или debug-state.

Результат:

- финальный lifecycle сессии становится прозрачным.

## Блок E. Fallback, escalation и timeout

### Задача 16. Поддержать команды пользователя

Что сделать:

- реализовать `отмена`;
- реализовать `начать заново`;
- реализовать `помощь`;
- определить, что происходит с session state и chat memory после каждой команды.

Результат:

- пользователь может управлять диалогом, не ломая intake-процесс.

### Задача 17. Добавить escalation / human review path

Что сделать:

- при высокой неоднозначности переводить сессию в `needs_human_review`;
- зафиксировать допустимые причины escalation;
- при желании использовать `send-and-wait` только как вспомогательный шаг подтверждения или ручного takeover;
- не заменять этим механизмом основной `Telegram Trigger`-паттерн.

Результат:

- у процесса появляется безопасный fallback для сложных случаев.

### Задача 18. Реализовать timeout policy

Что сделать:

- определить TTL для active session;
- обрабатывать заброшенные сессии;
- переводить их в `failed` или `cancelled`;
- не отправлять частичный callback без отдельного решения.

Результат:

- старые незавершенные диалоги не остаются навсегда активными.

## Блок F. Проверка и запуск

### Задача 19. Добавить тестовые сценарии на уровне workflow

Что сделать:

- проверить успешный start webhook;
- проверить создание сессии в Redis;
- проверить путь по каждому из 4 полей;
- проверить approximate answer;
- проверить final review;
- проверить успешный callback;
- проверить duplicate callback;
- проверить invalid `request_id` на стороне сервера;
- проверить cancel;
- проверить timeout;
- проверить escalation.

Результат:

- покрыт критический путь orchestration и conversational логики.

### Задача 20. Провести end-to-end smoke-check

Что сделать:

1. Отправить стартовый payload из сервера или mock-клиента.
2. Проверить ответ `202 Accepted`.
3. Проверить создание session state в Redis.
4. Проверить стартовое сообщение в Telegram.
5. Пройти диалог до конца.
6. Проверить payload callback на сервер.
7. Проверить, что сервер принял `202`.
8. Проверить, что сессия закрыта как `completed`.

Результат:

- подтвержден рабочий сценарий `server -> n8n -> Telegram -> server`.

## 6. Практический порядок выполнения

Если идти кратчайшим путем до рабочего решения, рекомендую такой порядок:

1. задачи `01-03`
2. задачи `04-07`
3. задачи `08-12`
4. задачи `13-15`
5. задачи `16-18`
6. задачи `19-20`

## 7. Контрольные точки

### Контрольная точка A

После задач `01-07` уже должен работать укороченный сценарий:

- сервер вызывает `n8n`;
- `n8n` отвечает `202`;
- создается session state;
- пользователю уходит стартовое сообщение.

### Контрольная точка B

После задач `08-12` уже должен работать живой intake-диалог без callback:

- Telegram-сообщения поднимают active session;
- AI-агент собирает все поля;
- формируется summary и подтверждение.

### Контрольная точка C

После задач `13-20` должен замкнуться полный сценарий:

`server -> n8n -> Telegram -> server`

## 8. Что получится в конце

После выполнения этого плана решение сможет:

- асинхронно принимать intake-команды от сервера;
- запускать и хранить Telegram intake-сессии;
- вести многошаговый AI-диалог с пользователем;
- сохранять детерминированное состояние разговора;
- корректно формировать callback в серверный DTO;
- обрабатывать retry, timeout, cancel и escalation;
- поддерживать прозрачный и проверяемый lifecycle каждой intake-сессии.
