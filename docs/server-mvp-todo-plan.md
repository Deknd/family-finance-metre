# TODO-план реализации MVP сервера v2

## 1. Назначение

Это вторая версия плана по доведению серверной части до MVP.

Документ собран по фактическому состоянию проекта на `17.03.2026` и
по актуальным документам:

- `docs/server-api-spec.md`
- `docs/server-collection-logic.md`
- `docs/database-spec.md`
- `docs/llm-agent-trigger-spec.md`
- `docs/device-ui.md`
- `docs/user-survey-mvp.md`

В отличие от первой версии, здесь перечислены только те задачи,
которые реально остались до MVP.

## 2. Что уже есть в проекте

На текущий момент в коде уже реализованы и покрыты тестами базовые части сервера:

- Spring Boot приложение на `Java 21` и `Spring Boot 4.0.3`;
- конфигурация приложения и properties для БД, security, `n8n` и scheduler;
- миграции `V1-V3` для справочных, payroll, интеграционных и snapshot-таблиц;
- локальный seed стартовой семьи и payroll-правил;
- JPA entity и repository для ключевых таблиц MVP;
- аутентификация по `X-API-Key` для `n8n`;
- аутентификация по `X-Device-Token` для устройства;
- `POST /api/v1/intake/user-finance-data`;
- валидация структуры payload;
- проверка ссылок `family_id` и `member_id`;
- защита от дублей по `external_submission_id`;
- поддержка optional `request_id` в intake payload;
- сохранение intake в `finance_submissions`;
- базовые unit и integration тесты для конфигурации, миграций, security, JPA и intake.

Это означает, что фундамент сервера уже собран.
До MVP теперь остаются не каркасные задачи, а замыкание бизнес-цепочки.

## 3. Что считаем готовым MVP

Сервер можно считать доведенным до MVP, когда он закрывает полный сценарий:

`server -> n8n -> Telegram -> server -> device`

Для этого должны одновременно работать такие возможности:

- сервер сам определяет payroll-события и запускает сбор данных;
- сервер создает и ведет `llm_collection_requests`;
- сервер вызывает webhook в `n8n` по контракту из `llm-agent-trigger-spec.md`;
- callback от `n8n` не только сохраняется в `finance_submissions`,
  но и пересчитывает `member_finance_snapshots` и `family_dashboard_snapshots`;
- устройство получает готовый dashboard через `GET /api/v1/device/dashboard`;
- сервер обновляет `devices.last_seen_at` после успешного ответа устройству;
- ключевые расчеты и интеграционные сценарии покрыты тестами;
- документация соответствует фактическому контракту.

## 5. Наблюдения по текущей документации

При планировании нужно учитывать несколько важных моментов:

- текущая документация уже фиксирует модель
  `один payroll-trigger = один intake payload`;
- `monthly_income` суммируется по всем intake за месяц,
  а `monthly_expenses`, `monthly_credit_payments` и `liquid_savings`
  берутся из самой свежей submission за период;
- device UI в MVP ожидает ровно один экран со статусом,
  четырьмя метриками и временем обновления;
- правила расчета `credit_load_percent` и `emergency_fund_months`
  и policy статусов `normal / warning / risk`
  теперь зафиксированы в документации и коде;
- в документации еще есть рассинхрон с фактической реализацией:
  проект уже использует UUID в контракте intake и уже умеет проверять `X-Device-Token`,
  поэтому финальная синхронизация docs остается обязательной задачей.

Статус: выполнено.

Что сделать:

- выбрать все `finance_submissions` по `member_id + period_year + period_month`;
- суммировать `monthly_income` по всем submission за период;
