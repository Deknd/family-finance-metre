# TODO-план реализации MVP сервера

## 1. Назначение

Этот документ описывает актуальный список задач, которые нужно выполнить,
чтобы довести текущую серверную реализацию до рабочего MVP.

План построен так, чтобы:

- каждая задача закрывала один конкретный шаг;
- задачи можно было брать в работу последовательно;
- на выходе сервер покрывал полный MVP-сценарий:
  `server -> n8n -> Telegram -> server -> device`.

## 2. Что уже есть в проекте

На текущий момент в проекте уже реализованы:

- базовый Spring Boot каркас;
- конфигурация окружения и profiles;
- миграции и seed справочных данных;
- JPA entity и repository для основных таблиц;
- простая аутентификация для устройства и `n8n`;
- endpoint приема данных `POST /api/v1/intake/user-finance-data`;
- валидация payload;
- защита от дублей по `external_submission_id`;
- сохранение intake данных в `finance_submissions`;
- базовое тестовое покрытие для конфигурации, миграций, security и intake.

Это значит, что фундамент приложения уже собран,
и дальше нужно закрыть именно бизнес-цепочку до полноценного MVP.

## 3. Что должно получиться после выполнения плана

После выполнения всех задач сервер должен уметь:

- принимать финансовые данные от `n8n`;
- обновлять персональные месячные snapshot'ы;
- пересчитывать семейный dashboard;
- отдавать готовые данные физическому устройству;
- вычислять зарплатные события;
- автоматически запускать `n8n` после нужных payroll-событий;
- связывать запуск опроса с последующим intake;
- хранить прозрачный статус всей цепочки обработки.

## 4. Границы этого TODO-плана

В этот план входят:

- замыкание intake в `member_finance_snapshots` и `family_dashboard_snapshots`;
- endpoint для физического устройства;
- обновление `last_seen_at`;
- расчет payroll-событий;
- scheduler запуска опросов;
- вызов webhook в `n8n`;
- связка `llm_collection_requests` с intake;
- тесты и обновление документации.

В этот план не входят:

- UI админки;
- публичные CRUD endpoint'ы для семей, участников и устройств;
- сложные финансовые модели и расширенная аналитика;
- праздничные календари;
- повторные напоминания пользователю;
- ручное редактирование финансов через веб-интерфейс.

## 5. Очередность задач

## Задача 01. Добавить доменную валидацию intake payload

Что сделать:

- проверять, что `family_id` существует;
- проверять, что `member_id` существует;
- проверять, что участник действительно принадлежит указанной семье;
- вернуть понятную ошибку в формате `422 VALIDATION_ERROR`, если ссылка невалидна.

Результат:

- сервер не сохраняет логически некорректные intake payload;
- ошибки по ссылкам перестают превращаться в внутренние ошибки сервера.

## Задача 02. Расширить intake-контракт optional полем `request_id`

Что сделать:

- добавить в request DTO необязательное поле `request_id`;
- сохранить обратную совместимость со старым payload;
- использовать это поле только если оно реально пришло в запросе.

Результат:

- intake endpoint готов к сквозной связке с ранее созданным `llm_collection_request`.

## Задача 03. Реализовать upsert в `member_finance_snapshots`

Что сделать:

- по `member_id + period_year + period_month` создавать или обновлять snapshot;
- брать данные из уже сохраненного `finance_submission`;
- обновлять snapshot только если новый `collected_at` свежее текущего;
- сохранять ссылку на `source_submission_id`.

Результат:

- у сервера всегда есть актуальный месячный срез по каждому участнику семьи.

## Задача 04. Реализовать сервис агрегации семейных показателей

Что сделать:

- собрать все `member_finance_snapshots` семьи за нужный период;
- суммировать доходы;
- суммировать расходы;
- суммировать кредитные платежи;
- суммировать накопления;
- посчитать `member_count_used`.

Результат:

- сервер умеет собрать семейный срез из персональных snapshot'ов.

## Задача 05. Реализовать сервис расчета dashboard-метрик

Что сделать:

- посчитать `credit_load_percent`;
- посчитать `emergency_fund_months`;
- использовать `BigDecimal` и округление `HALF_UP`;
- обработать edge case'ы с нулевым доходом и нулевыми расходами.

Результат:

- агрегированные семейные значения готовы для сохранения и показа на устройстве.

## Задача 06. Реализовать policy расчета статуса семьи

Что сделать:

- определить правила для `normal`, `warning`, `risk`;
- вычислять `status_text`;
- вычислять `status_reason`;
- держать правила в отдельном policy/service классе.

Результат:

- сервер отдает не только цифры, но и понятный итоговый статус для UI.

## Задача 07. Реализовать сохранение `family_dashboard_snapshots`

Что сделать:

- после intake пересчитывать dashboard семьи за соответствующий период;
- создавать или обновлять `family_dashboard_snapshot`;
- сохранять расчетные поля и `calculated_at`.

Результат:

- у сервера появляется готовый snapshot для выдачи физическому устройству.

## Задача 08. Собрать intake orchestration в одной транзакции

Что сделать:

- сохранить `finance_submission`;
- обновить `member_finance_snapshot`;
- пересчитать семейные метрики;
- сохранить `family_dashboard_snapshot`;
- при наличии `request_id` подготовить связку с `llm_collection_request`.

Результат:

- intake поток становится полноценной бизнес-операцией, а не только сохранением сырого события.

## Задача 09. Реализовать сервис чтения dashboard для устройства

Что сделать:

- по устройству находить связанную семью;
- выбирать последний готовый `family_dashboard_snapshot`;
- готовить модель ответа для device endpoint.

Результат:

- у сервера появляется прикладной сервис выдачи dashboard на экран устройства.

## Задача 10. Реализовать `GET /api/v1/device/dashboard`

Что сделать:

- поднять endpoint для физического устройства;
- брать устройство из уже выполненной аутентификации;
- вернуть структуру ответа из `server-api-spec.md`;
- возвращать `404`, если dashboard пока отсутствует.

Результат:

- устройство может получить готовые данные для отображения.

## Задача 11. Добавить обновление `devices.last_seen_at`

Что сделать:

- обновлять `last_seen_at` после успешного ответа устройства;
- не обновлять `last_seen_at`, если запрос завершился ошибкой.

Результат:

- сервер знает, когда устройство последний раз было на связи.

## Задача 12. Добавить mapping ошибок для device/dashboard и доменных ссылок

Что сделать:

- добавить единый ответ для случая `DASHBOARD_NOT_READY`;
- добавить читаемые ошибки для несуществующей семьи;
- добавить читаемые ошибки для несуществующего участника;
- добавить читаемые ошибки для mismatch `member -> family`.

Результат:

- API отдает стабильный и понятный формат ошибок по всем основным сценариям MVP.

## Задача 13. Реализовать сервис расчета payroll-событий

Что сделать:

- считать номинальную дату выплаты;
- считать фактическую дату выплаты;
- поддержать `fixed_day_of_month`;
- поддержать `last_day_of_month`;
- переносить субботу на пятницу, воскресенье на понедельник;
- считать `scheduled_trigger_date`.

Результат:

- сервер умеет определить, когда именно нужно запускать опрос пользователя.

## Задача 14. Реализовать ежедневный scheduler запуска опросов

Что сделать:

- включить планировщик;
- запускать ежедневную job;
- проходить по активным `member_payroll_schedules`;
- учитывать timezone семьи;
- определять, для кого сегодня должен стартовать опрос.

Результат:

- сервер автоматически находит события, по которым надо инициировать сбор данных.

## Задача 15. Реализовать создание `llm_collection_requests`

Что сделать:

- создавать запись перед вызовом `n8n`;
- сохранять period, payroll даты, reason и requested fields;
- защититься от дублей по одному payroll-событию;
- хранить `request_payload`.

Результат:

- каждый запуск опроса становится трассируемым и идемпотентным.

## Задача 16. Реализовать HTTP-клиент для вызова `n8n`

Что сделать:

- собрать payload по `llm-agent-trigger-spec.md`;
- вызывать webhook через `RestClient`;
- передавать bearer token;
- поддержать timeout;
- сохранять `response_payload` или ошибку вызова.

Результат:

- сервер умеет реально инициировать работу LLM-агента через `n8n`.

## Задача 17. Реализовать lifecycle статусов `llm_collection_requests`

Что сделать:

- при успешном ответе `n8n` переводить запись в `accepted`;
- сохранять `workflow_run_id`;
- при ошибке переводить запись в `failed`;
- сохранять `error_message`.

Результат:

- у сервера появляется прозрачный статус по каждому запуску опроса.

## Задача 18. Реализовать связку intake с `llm_collection_requests`

Что сделать:

- при intake искать `llm_collection_request` по `request_id`;
- связывать с ним `finance_submission`;
- после успешного обновления snapshot'ов переводить запрос в `completed`;
- сохранять `completed_at`.

Результат:

- цепочка `server -> n8n -> Telegram -> server` становится сквозной и наблюдаемой.
